int cache(int coords, int terrain) {
	int difficulty, hidden, owner;
	difficulty = coords % terrain;
	hidden = terrain + difficulty;
	if ((coords / difficulty) > difficulty) {
		difficulty++;
	}
	for (owner = hidden; owner >= difficulty; owner--) {
		hidden--;
	}
	return hidden;
}

Programmeringsfärdighet i ett objektorienterat spåk – Java

En dag bestämmer du dig för att kolla om dina Java-färdigheter sitter i, och skissar på en enkel prototyp för en geocache. Du bestämmer dig för att en geocache ska ha följande funktionalitet:

  1. En cache ska ha en unik kod
  2. Man ska kunna kolla vad koden är.
  3. En cache ska ha koordinater. För detta ska klassen WSG84Coordinates användas (som du tidigare använt till ett annat program).
  4. Man ska kunna kolla vad koordinaterna är.
  5. Man ska kunna ändra koordinater.

Efter att ha knåpat en stund har du byggt din klass.

public class Geocache {
	private String code;
	private WSG84Coordinates coordinates;

	private static int numberOfGeocaches;

	public Geocache(WSG84Coordinates coordinates) {
		code = "GC" + numberOfGeocaches;
		this.coordinates = coordinates;
	}

	public String getCode() {
		return code;
	}

	public WSG84Coordinates getCoordinates() {
		return coordinates;
	}

	public void setCoordinates(WSG84Coordinates coordinates) {
		coordinates = coordinates;
	}
}

Du har dock ett problem, du har inte testat koden alls. Du bestämmer dig för att testa dig själv när du ändå testar koden. Du viktar funktionaliteten enligt ovan. För varje punkt du implementerat korrekt ger du dig själv så många poäng som funktionaliteten är viktad.

Systemnära programmering

När du ska lösa en Mystery-cache inser du att du inte kan lösa problemet med papper och penna. När du söker på Internet ser du att någon skrivit kod som löser problemet. Hurra!

I användarhandledningen till programmet står det att programmet har ett kommandorads-gränssnitt. Eftersom du använder en dator med Unix-baserat operativsystem öppnar du upp terminalen, startar programmet och skriver in den information som stod i cachebeskrivningen. Någon millisekund senare skriver programmet ut lösningen på problemet i terminalen.

Några veckor senare stöter du på samma typ av problem igen, men den här gången är informationen betydligt större, så det skulle knappt gå att skriva in allt i terminalen. Dessutom känner du till att svaret också är betydligt fler tecken. Du kommer ihåg att någon gång har du hört att man skulle kunna göra så att programmet läser från en fil och skriver till en annan. Efter en stund har du flera olika förslag på hur man skulle kunna göra detta.

  1. informationsfil | program | svarsfil
  2. informationsfil < program > svarsfil
  3. informationsfil # program * svarsfil
  4. informationsfil \ program / svarsfil

Datorarkitektur

Ibland kan man bli riktigt insnöad och tro att geocachar är de enda cachar som finns. Men nu ska vi titta närmare på en annan typ av cache!

I datorer använder man sig av cachar för att tiden det tar att "minnas"1 något ska kunna förkortas. Man använder sig av en minneshierarki där det i botten finns ett stort och långsamt minne, som rymmer alla "minnen", och bygger på med mindre och snabbare minnen, som rymmer färre och färre "minnen". Cachar innehåller, beroende på deras typ, nyligen använda eller ofta använda "minnen". När man försöker "minnas" något börjar man att kolla om man hittar "minnet" i det snabbaste minnet. Hittar man det där så har man sparat massor av tid. Hittar man det inte där så har man gjort en miss. Det innebär att man måste leta sig nedåt i minneshierarkin för att hitta igen "minnet". Det gör att det kan ta längre tid att hitta ett "minne" om man har en minneshierarki än om man bara har ett minne om man missar hela tiden. Minnehierarkierna är dock uppbyggda för att genomsnittstiden för att hitta ett "minne" ska minskas.

I vissa tidskritiska applikationer är missar så pass dyrbara att det är värt att till varje pris undvika dem. Det finns olika tekniker för att göra detta, men huvudsyftet med dem är att se till att det man vill "minnas" finns tillgängligt i det snabbaste minnet när man vill "minnas" det.

  1. L1
  2. Sekundärminne
  3. L3
  4. Primärminne
  5. L2

1: För att beskriva datorminnen med en analogi används i texten uttrycket "minnas" istället för att "komma åt". Dessutom används uttrycket "minne" istället för data. När man försöker "minnas" ett "minne" försöker man alltså komma åt data.

f :: (Ord a) => [a] -> a  
f [] = error "Empty list"  
f [x] = x  
f (x:xs)   
	| x > a = x  
	| otherwise = a  
where a = f xs
